Sometimes folks joining the DDMLab are experienced programmers for statistics and modeling, but with no experience writing web applications, but are suddenly thrust in a rôle of having to do just that. Here are some notes jotted down for the help of one such person that may be of at least a little help to others.

First, a couple of bits of general advice. Much, probably most, of what you might find through Google and so on about web programming is aimed at people building web sites, generally ones trying to sell things, or at least trying to be popular. This is, of course, a bit different than what we’re trying to do. In particular
* Whenever they’re worrying about things that make stuff scale better to many thousands of users, we probably don’t care. A couple of dozen users is probably the most we expect to be banging on things simultaneously.
* Whenever they’re worrying about supporting all sorts of old or obscure browsers, and stuff like that, we can ignore it. It’s reasonable for us to only expect users who have recent, mainstream browsers.
* We couldn’t care less about how easily folks can find us with search engines or how far we are up the Google page rankings—typically we **DON’T** even **WANT** folks to find us that way!

Next, I’d suggest making sure you have some understanding of how the web works. For this stuff the defining documents are a bunch of RFCs (Requests For Comment) published by the IETF (Internet Engineering Task Force). Confusingly these don’t get updated, they get replaced by newer versions, and sometimes you need to have to look at multiples of them. Oh, and some of them, typically those published on 1 April, are adoxographic. Anyway, they’re basically standardsy legalese and you needn’t look at them unless you really, really care about some fine point. Better to go looking at someone’s more human readable summary. I generally find the tutorial-ish stuff published on MDN (the Mozilla Developer Network, https://developer.mozilla.org/en-US/) both the most helpful, and most accurate; that’s my usual first place to try to learn this stuff, and usually it doesn’t disappoint.

But here’s a brief adumbration of the central protocol of it all. Circa thirty years ago the HTTP (Hypertext Transfer Protocol; HTTPS is essentially the same thing, just with a bunch of added complications ensuring the traffic is encrypted) was created, defining the web. It’s not important to know the low level, networky details of it (which have evolved, and continue to do so), but you need to know its overall shape.

The most important thing to know about it is that it’s kind of a one-way thing. A client (“user agent,” is, I think, the official term; typically a web browser) initiates a transaction with a server, and the later returns something to the client. That’s it. The server CANNOT initiate anything.

There are a bunch of different kinds of requests the client can make of the server, but the only two you probably have to worry about are GET and POST. These are actually pretty similar, the main difference being simply that some ancillary data (“request parameters”) you can send along with the request are sent in different ways. With GET they’re part of the URL, with POST they go under the covers. As part of the URL means you can kind of encapsulate it all into one URL (that “?…=…” stuff I’m sure you’ve seen) which is sometimes good, sometimes not so; but it also means there’s a tight limit on how much you can send, and it can only be characters that can go into the URL, so POST is more often useful once you've got into the app.

When the server responds it sends various bits of meta data along with the real stuff.  The most important are what kind of stuff it’s sending back, typically HTML (Hypertext Markup Language) and a status code. These latter are three digit numbers. Typically “200 OK” which means it's sent stuff back and all is good. But sometimes an error, most commonly “404 Not Found” or “50n …” for some digit n which means the server messed up.

One of the other things to bear in mind is that HTTP typically has other entities, besides the server and client, looking at the traffic as it goes by, and possibly doing things with it, typically caching stuff. This is mostly done to help folks with scalability, so we don’t need to know much about it. But we need to know it’s there, as it sometimes gets in our way and causes confusion, typically while debugging. Shift-refresh is your friend in the browser as it typically clears the browser’s own cache, which is a common source of confusion when developing and debugging stuff.

The one bit of lower level, network stuff you need to know is that HTTP is normally implemented on top of the lower level network protocol TCP/IP, and you typically get at servers talking TCP/IP by connecting to a “port,” ports being named by non-negative integers. The ones below 1,024 are typically dedicated to special uses. 80 is for HTTP and 443 for HTTPS, and a server will typically have a mainstream web server of some sort running on both of them. For URLs of the form http://mumble.cmu.edu/foo or whatever that’ll go to port 80; but you can go to a different port with a URL like http://mumble.cmu.edu:8088/foo.

 When we write something we typically pick a high numbered port (in the range 1,024 to maybe 20,000) that we’ll use. While we’ve never had a problem with different people in the group picking the same port number we inevitably will if we don’t coördinate, so we have a little spreadsheet for that purpose. And then we just use one of those URLs with a port number to access it. But you *can* arrange to have the web server on port 80 pipe stuff to your thing, and so not need to expose the magic port number (the term of art is “reverse proxy”). This is a little more bother since (a) you need to adjust the web server’s configuration, and (b) need to pick your URLs appropriately so it can direct the traffic whence it should go; and, in any case, you still need to pick a high number port for your own app to communicate with the web server with some protocol or other. It’s rarely worth the trouble for us, but might be necessary if you need to use HTTPS.

Anyway, if all this isn’t old hat, I suggest learning a little about it. MDN is a good place to look.

Armed with this knowledge, you can figure out if a straightforward, old fashioned web like thing is good enough for your purposes. If it is, you’re in great shape: you can use pretty much any programming language you like (OK, probably not COBOL, APL or Prolog) to write the server-side stuff, and you don’t have to write any code that runs in the browser. And your life will be simpler in many other ways.

But remember, this means everything has to start with the client saying “give me something,” and your server code giving it to them, and then largely washing their hands of it until the client chooses to ask for something else, probably sending something back with it.

A good example of something that works this way is implementing a simple binary choice game. The user says “hey, I want to play.” You send back the choices, which include buttons they can click, A or B. When they click one, it again sends something back to the server saying “I picked A” or “I picked B,” you work out what the result is and send it back to them, along with another button they click to play the next round.

But what you can't do with vanilla HTTP is say “you have to make your choice in the next twenty seconds or I’m going to make it for you and tell you what it is then.” That would require the server being able to send a second, overriding response back to the client. If you need to do something like this, you’re going to need more complex technology. More on that below.

Next, whether you need to do it the easy way or the hard way, you’re almost certainly going to have to learn something about HTML. This is the stuff with <angle> </brackets> I’m sure you’ve seen. It’s basically a hierarchical description of how to layout stuff on a page. It’s typically the guts of what a server sends back to a client. MDN is a great resource for learning about it. It, too, has gone through a bunch of revisions. I’d stick with just the current version, HTML5. Anybody starts to try to tell you about XHTML or HTML4 or the weird differences between versions of IE (all of which are now obsolete anyway), I’d just skip it and move on. For the HTML stuff the bits you almost certainly need to know about are the basic structure of documents, and forms which are how the user does things like telling you “I clicked button B” or “I want to share $3.50 of my alloted money.”

An important adjunct to HTML is CSS (Cascading Style Sheets). Again, MDN is the place to look. This is where you typically say what font to use, whether italics, and, most importantly, how big things need to be. In older versions of HTML this stuff tended to be shoved into the HTML instead of CSS, but it was a mess, and just gets you into trouble. It’s a pain, but you’ll be doing yourself a favor if you learn at least the basics about CSS.

One of the issues HTML(+CSS) is dealing with is that every user potentially has a different sized screen with potentially different capabilities (resolution, colors, etc), but you’re sending one description that has to work on all of them. For our purposes we can frequently limit things a lot (e.g. the end user is required to use a computer monitor, not a mobile phone), but even with that, the technology is designed for more variability so you at least have to understand it a little bit.

IF you need something more complicated than basic HTTP type stuff, you’re going to have to learn JavaScript (aka ECMAScript, and often abbreviated “JS”). Despite its name it has nothing to do with Java. The name is just a bit of marketing stupidity perpetrated by the boss of the guy who designed it, because when it was being designed Java was the hot cool new thing. It’s really Scheme (a Lisp dialect) wrapped up in a C-like syntax instead of the far more convenient parentheses :-) If you need to learn about JavaScript, a good place is (surprise!) MDN.

The reason you’ll need JavaScript in this case is because you’ll almost certainly then have to run at least a little code actually in the browser (perhaps just a timer for twenty seconds if you’re limiting the subject to only that long to make a choice). And JavaScript’s pretty much the only game in town for that these days.

If you need a lot of JS there are libraries you can use (you cite them in the <HEAD> part of your HTML) that simplify things. The 800 pound gorilla in this space is jQuery. For this the goto learning place is not MDN, it’s the jQuery web site: https://api.jquery.com/. If you’re writing a lot of JS for your web application, it's a big help, but unless you really need it, I’d probably not bother.

Note that you can also use JavaScript to write your server, typically running it with node.js. But since you already know Python, that’s probably not a good choice for you. But it is a common choice for many other folks. And if you end up doing stuff with nodeGame, which is widely used in the DDMLab, it IS all written in JS, even on the server side.

OK, so far we’ve just been talking about the abstract technologies that will be used to get a server and a client talking. What about the actual code on the server side? There are bazillions of possibilities, made even more complicated as they’re often used rather like ordering a pizza, with different things laid on top of others. Besides coming in different languages, they also come at different points along the spectrum from “we just supply a bunch of legos you can assemble to your own taste to get things done” to “we have a very strong opinion about how things should be done and you pretty much must just do it our way, but if your problem does fit our mental model closely there will be very little you actually have to write.” My own experience has taught me to favor the former, but YMMV.

Anyway, for Python the three leading contenders are, I think, Django, Bottle and Flask.

Django is huge, complex, and at the Soup Nazi end of the spectrum. It’s really for building big web sites, and not a particularly good fit for the stuff we do, but I think I remember someone has used it. There’s a lot to be learned to use it, but it’s well documented: https://docs.djangoproject.com/en/4.0/

I’ve never used Bottle, but it has a reputation for simplicity. I’ve also heard it alleged that the documentation is disappointing, but here it is: https://bottlepy.org/docs/dev/.

Were Ι building something in Python, I’d look first at Flask. I’ve built stuff on top of it in the past and found it reasonably convenient. It’s pretty light weight, and not too horribly opinionated. It’s also well documented: https://flask.palletsprojects.com/en/2.0.x/. There are also a host of third party tutorials Dr Google will happily tell you about.

There are innumerable other choices, many of which are kind of creaky and ill-supported.

Getting vanilla code to emit the big, wordy, nested stuff that is HTML can be a pain. The usual route is, besides the server stuff per se, to use a template language of some sort. As usual, there are a bazillion of them. Basically you write a file that looks like the page you’re trying to emit but that has placeholders for things your code has to figure out and plug in (as an aside, one of the joys of writing Lisp is that this kind of thing is *much* easier, and doesn’t require all these darn template files :-) but if you’re in Python, you'll probably want a template thingie).

By default Flask uses the Jinja (https://jinja.palletsprojects.com/en/3.0.x/) template system, though you can probably plug other ones in if you really want to. Other systems will typically have their own preferred template systems, but again alternatives can usually be plugged in to taste. When in the past I’ve built Flask apps Jinja has seemed just fine. Besides the Jinja doc, the Flask doc also has a good description of how to write Jinja templates.

If you need more than just the simple client-pokes-the-server-and-gets-back-a-response stuff you’ll likely want to look into a technology, or really ensemble of technologies, usually called AJAX (Asynchronous Java and XML, though these days generally used with JSON rather than XML—JSON is a simple little data description language that is closely tied to JavaScript). The heart of this is an extension to HTTP called XMLHttpRequest. You set up a web page with the usual HTML(+CSS), but instead of replacing it, you adjust it by sending tiny bits of information back and forth between the client and the server. This typically means having a lot of JS code in the client, as well as probably using CSS extensively. For example, this is (Ι believe) how the GMail web client works. It’s good for stuff like games where you expect the user to navigate around (a la Minecraft) by pressing a key several times each second—for something like this you have to do a lot of the work in the client, ’cause it would take to long to have a bunch of round trips to the server and back. This is way more complicated so I’d avoid it unless you really need it. If you do need it the best starting place to learn are, I think, MDN and the relevant jQuery doc.

There are innumerable libraries, both client and server, that try to help you do this kind of stuff. Each has its own learning cliff, and pluses and minuses. There are also cool, related technologies that try to simplify this stuff. If really needed they can be a help. One I’ve used is WebSocket, an alternative to HTTP that allows continuously connected clients and servers. MDN has documentation on the protocol, and there are a bazillion implementations of servers for building the stuff. I’d avoid it unless you really need it, though.

Databases.

Most people have a database behind their web app, usually a relational one. What it’s used for varies. For complex websites (especially those built with Django) it is used, among other things, for holding descriptions of what the site looks like. That is almost certainly more than we need. We mostly just need a database (if we need one at all) to keep the results of what users did.

But it’s kind of a superstition that this *has* be a database. If your needs are simple enough, you can probably just write out text files, most likely CSV files. This is a little complicated by the fact that if you have two or more threads or processes writing a single text file, it’s going to turn into a mess unless you coördinate their activities. But if you have each process/thread write to its own uniquely named file, all will be well, and you just need to concatenate them when you want to analyze the results.

If you do decide you need to use a database, the most common choice seems to be MySQL (or MariaDB—one’s a fork of the other, but mostly they’re the same thing) (https://www.mysql.com/). Though I tend to prefer SQLite (https://sqlite.org/index.html) as I find it a lot simpler. MySQL is a server itself, which is running on Halle (and will be on the new machine whenever it arrives), so you need to create a database in it using the mysql client. For SQLite a database is just a file, and it pretty much gets created as needed (actually, one of the common error modes is if you try to open a DB that doesn’t already exist it tries to create it for you, which can be miserably confusing! There are flags you can supply that say “don’t do that,” which I recommend!).

In either case, if you need to use a database you’re going to have to learn SQL (Structured Query Language) for writing to it, and possibly for reading from it. And, possibily, an introduction to what a relational database is at all. One good starting place seems to be in the MariaDB documentation: https://mariadb.com/kb/en/introduction-to-relational-databases/. Once you feel you know what’s going on, the documentation for MySQL or SQLite will tell you details and provide reference for the particular dialect of SQL they support (every DB is a little different in that regard, though the basics pretty much remain the same). If you need not just to write to a database, but to read from it to build up more complex things to write, be sure to use transactions to keep stuff from getting into a muddle. But the simpler you can keep things, the happier you’ll be.

Warning: if for some reason you choose to go the Django route, while it will be built on top of a database (probably MySQL), you don’t write the SQL code directly with Django, instead you use their fancy ORM (Object Relational Mapper) to make things “easier”—where easier means less code to write but more stuff to learn, and you have to do it their way whether you want to or not.

Whew! Hope that helps.
